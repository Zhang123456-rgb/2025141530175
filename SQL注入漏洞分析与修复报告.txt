一、注入漏洞来源分析

 什么是注入

注入（

 ）是

中危害最大的漏洞之一。攻击者在输入框插入恶意代码，利用后端对用户输入的不当处理，使恶意被执行。根本原因

是使用字符串拼接而非参数化查询。

 漏洞代码定位

漏洞点：搜索功能（ 和 ）

  # VULNERABLE: f-string concatenation with user input

  keyword = request.args.get("keyword", "")

  sql = f"SELECT id, username, email, phone FROM users

         WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"

  c.execute(sql)    # UNSAFE!

漏洞点：注册功能（）

  sql = f"INSERT INTO users (username, password, email, phone)

         VALUES ('{username}', '{password}', '{email}', '{phone}')"

  c.execute(sql)    # UNSAFE!

危险：用户输入直接嵌入语句，无任何过滤或转义！

三、漏洞危害分析

本次发现的注入漏洞具有以下危害：

 数据泄露：  可获取所有字段（含密码）

 绕过认证：  永真条件绕过数据访问控制

 数据篡改：注册注入点可修改或删除数据

 隐私泄露：邮箱、手机号泄露可能导致钓鱼攻击

 权限提升：注入在某些配置下可执行系统命令

根因：

使用  直接拼接用户输入到  语句

没有使用参数化查询（ ）

没有对用户输入做任何过滤或转义

四、修复方案

 参数化查询原理

参数化查询将结构和数据分离。数据库驱动自动转义和绑定参数，用户输入永远无法改变语义。

  # SAFE: Use ? placeholders + parameter tuple

  sql = "SELECT * FROM users WHERE username LIKE ? OR email LIKE ?"

  params = (f"%{keyword}%", f"%{keyword}%")

  c.execute(sql, params)  # SAFE!

  # Even if keyword is "%' OR 1=1 --"

  # DB driver treats it as plain text, NOT SQL

 修复前后代码对比

搜索功能修复

  # BEFORE (VULNERABLE) - f-string

  sql = f"SELECT ... WHERE username LIKE '%{keyword}%'"

  c.execute(sql)

  # AFTER (FIXED) - 

  sql = "SELECT ... WHERE username LIKE ? OR email LIKE ?"

  params = (f"%{keyword}%", f"%{keyword}%")

  c.execute(sql, params)

注册功能修复

  # BEFORE (VULNERABLE) - f-string

  sql = f"INSERT INTO users VALUES ('{username}', ...)"

  c.execute(sql)

  # AFTER (FIXED) - 

  sql = "INSERT INTO users VALUES (?, ?, ?, ?)"

  params = (username, password, email, phone)

  c.execute(sql, params)
